## Что работает сейчас

### 5 микросервисов (docker compose up -d)

| Сервис | Технология | Порт | Роль |
|---|---|---|---|
| **parser** | Python / FastAPI | 8001 | Читает XML-выгрузку конфигурации 1С (36 типов объектов) |
| **embeddings** | Python / BERTA (768d) | 5050 | Векторизация описаний и кода (dense + sparse BM25) |

| **mcp-server** | TypeScript / Node.js | 8000 | HTTP MCP-сервер, **9 инструментов** |

Все 5 сервисов healthy, перезапускаются автоматически.

### 9 MCP-инструментов

#### Для аналитика 1С

| Инструмент | Что делает | Пример |
|---|---|---|
| `1c_metadata_search` | Найти объект по описанию на русском | «контрагенты», «приходная накладная» |
| `1c_metadata_details` | Структура объекта: реквизиты, табличные части, движения | Детали справочника «Номенклатура» |
| `1c_metadata_types` | Сколько объектов каждого типа | 98 справочников, 124 документа... |
| `1c_subsystems` | В каких бизнес-модулях находится объект | «Номенклатура» → 27 подсистем |
| `1c_odata_query` | Запрос к живой базе 1С через OData | Получить список контрагентов |
| `1c_register_balances` | Остатки и обороты регистров | Остатки ТМЦ на складе |
| `1c_register_movements` | Движения регистров за период | Движения за март 2026 |

**Типичный сценарий аналитика:**
1. `1c_metadata_search` → найти нужный объект
2. `1c_odata_query` / `1c_register_balances` → получить данные из живой базы
3. Анализ и выводы

#### Для разработчика 1С

| Инструмент | Что делает | Пример |
|---|---|---|
| Все 7 выше | + | + |
| `1c_code_search` | Поиск по BSL-коду: процедуры, функции | «ОбработкаПроведения» в конкретном документе |
| `1c_dependencies` | Граф зависимостей: документ ↔ регистры | «ДвижениеТМЦ» ← 24 документа |

**Типичный сценарий разработчика:**
1. `1c_metadata_search` → найти объект
2. `1c_dependencies` → понять, что затронется при изменении
3. `1c_subsystems(find)` → понять бизнес-роль объекта
4. `1c_metadata_details` → получить структуру (реквизиты, ТЧ)
5. `1c_code_search` → найти существующий код
6. Написать код с полным контекстом

| `1c_syntaxcheck_mcp` | 8002 | Разработчик | Проверка синтаксиса BSL |
| `mcp_ssl_server` | 8008 | Разработчик | Справка по БСП (Библиотека стандартных подсистем) |

**Для запуска:**
```bash
cp .env.example.comol .env.comol
# заполнить ключи в .env.comol
docker compose -f docker-compose.comol.yml --env-file .env.comol up -d
```

## Использовали ли мы дистрибутивы comol (MCP Distr)?

**Нет, мы их НЕ использовали.** Образы comol прописаны в `docker-compose.comol.yml`, но ни разу не запускались. Лицензионные ключи не введены.

Всё, что сейчас работает — **написано нами с нуля:**

| Что написали сами | Аналог в comol |
|---|---|
| `1c_metadata_search` (гибридный поиск BERTA+BM25) | `1c_code_metadata_mcp` (ChromaDB) — дублирует |
| `1c_code_search` (поиск по BSL-коду) | `1c_code_metadata_mcp` — дублирует |
| `1c_dependencies` (граф из Qdrant, без Neo4j) | `1c_graph_metadata` (Neo4j + 2GB + OpenAI) — тяжелее |
| `1c_subsystems` (навигация по подсистемам) | **Нет аналога ни у кого** |
| `1c_odata_query` + `1c_register_balances` + `1c_register_movements` | **Нет аналога ни у кого** |
| Скилл `1c-analyst` | **Нет аналога ни у кого** |

**Уникальные фичи (только у нас):**
- OData-интеграция (3 инструмента для живых данных из 1С)
- Навигация по подсистемам
- Аналитический скилл с 50+ паттернами анализа
