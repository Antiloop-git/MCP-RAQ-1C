# Аналитик 1С: от бизнес-вопроса к инсайту

Полностью автономный скилл для аналитики и запросов к базам 1С:Предприятие. Включает бизнес-анализ, справочник синтаксиса запросов и выполнение.

## Главные принципы

1. **От вопроса к данным.** Сначала пойми, что хочет узнать пользователь, потом ищи нужные объекты.
2. **Никогда не угадывай имена объектов.** Имена в 1С — русскоязычные и специфичные для конфигурации. Всегда используй `1c_metadata_search` → `1c_metadata_details` перед написанием запроса.
3. **Регистры важнее документов.** Для аналитики данные лежат в регистрах (агрегированные, с измерениями и ресурсами). Документы — только для расследования конкретных операций.
4. **Инкрементальный подход.** Начинай с `ПЕРВЫЕ 10` — убедись что структура верна, потом расширяй.

## Инструментарий

| Инструмент | Роль в аналитике |
|------------|-----------------|
| `1c_metadata_search` | Найти регистры/справочники по бизнес-вопросу |
| `1c_metadata_details` | Понять структуру: измерения, ресурсы, реквизиты |
| `1c_metadata_types` | Обзор конфигурации — сколько объектов каждого типа |
| `1c_code_search` | Найти бизнес-логику в коде (ценообразование, скидки, проведение) |
| `1c_odata_query` | Быстрые запросы через OData (простые выборки, интеграция) |
| `1c_register_balances` | Остатки регистров + автодетекция отрицательных |
| `1c_register_movements` | Движения регистра — расследование конкретных операций |
| `query-1c.sh` / `.ps1` | Сложные запросы: соединения, агрегация, виртуальные таблицы |

**Когда что использовать:**
- **OData-инструменты** (`1c_odata_query`, `1c_register_balances`, `1c_register_movements`) — быстрые проверки, простые выборки, интеграционные запросы
- **query-1c.sh** — сложная аналитика с JOIN, GROUP BY, подзапросами, виртуальными таблицами с параметрами

### Доступные OData-сущности (ASTOR Prod)

**Справочники:**
| Entity Set | Описание |
|-----------|----------|
| `Catalog_Номенклатура` | Товары, продукция, ингредиенты |
| `Catalog_Склады` | Склады (+ таб. часть `_СкладыПроизводства`) |

| `Catalog_ЕдиницыИзмерения` | Единицы измерения |

**Регистры накопления:**
| Entity Set | Тип | Виртуальные таблицы |
|-----------|------|-------------------|
| `AccumulationRegister_УчетПартий` | **Остаточный** | `/Balance` (КоличествоBalance, СуммаБезНДСBalance, СуммаНДСBalance), `/Turnovers`, `/BalanceAndTurnovers` |
| `AccumulationRegister_ДвижениеТМЦ` | **Оборотный** | `/Turnovers` (КоличествоTurnover, СуммаTurnover, ПрибыльTurnover) |

**Документы:**
| Entity Set | Описание | Табличные части |
|-----------|----------|----------------|
| `Document_ПриходнаяНакладная` | Приход товаров | `_Состав`, `_РазделыГТД`, `_ЗаказыТоваров`, `_ДатыПроизводства` |
| `Document_РасходнаяНакладная` | Расход/реализация | `_Состав`, `_ДатыПроизводства`, `_ЗаказыТоваров` |
| `Document_Спецификация` | Спецификации | `_Состав`, `_ДатыГотовности` |
| `Document_Смолл_КалькуляционнаяКарта` | Техкарты/калькуляции | `_Состав`, `_ТехнологическиеКартыПолуфабрикатов` |

**Ключевые поля регистра УчетПартий:**
- Измерения: `Номенклатура_Key`, `Характеристика_Key`, `МестоХранения` (склад), `Фирма_Key`, `Статус`, `Партия_Key`
- Ресурсы Balance: `КоличествоBalance`, `СуммаБезНДСBalance`, `СуммаНДСBalance`, `СуммаРеализацииBalance`, `СуммаУРЦBalance`
- RecordType: `RecordType` = `Receipt` (приход) / `Expense` (расход)

**Важно:** Виртуальные таблицы вызываются через FunctionImport (путь с `/`), а не EntitySet (с `_`):
- Правильно: `AccumulationRegister_УчетПартий/Balance`
- Неправильно: `AccumulationRegister_УчетПартий_Balance`

## Универсальный аналитический workflow

### Шаг 1. Декомпозиция бизнес-вопроса

Разбей вопрос на компоненты:

| Компонент | Пример для «какие товары плохо продаются?» |
|-----------|-------------------------------------------|
| **Сущность** | Номенклатура (товары) |
| **Метрика** | Количество/сумма продаж |
| **Группировка** | По товару |
| **Фильтр** | За период, только элементы (не группы) |
| **Критерий** | Низкие продажи при наличии остатков |

### Шаг 2. Поиск объектов метаданных

Используй `1c_metadata_search` с ключевыми словами из предметной области:

| Домен | Ключевые слова для поиска |
|-------|--------------------------|
| Продажи | продажа, реализация, выручка, товарооборот |
| Склад | остаток, товар, склад, движение, приход, расход |
| Финансы | взаиморасчет, задолженность, оплата, долг |
| Цены | цена, прайс, скидка, наценка |
| Закупки | поступление, поставщик, заказ поставщику |
| Производство | техкарта, спецификация, рецептура, состав |

Ищи в первую очередь **регистры** (AccumulationRegister, InformationRegister) — они хранят агрегированные данные.

### Шаг 3. Изучение структуры

Вызови `1c_metadata_details` для найденных объектов. Обрати внимание на:
- **Измерения** регистра → поля для GROUP BY и фильтров
- **Ресурсы** регистра → поля для SUM/агрегации
- **Виртуальные таблицы** → Остатки(), Обороты(), ОстаткиИОбороты(), СрезПоследних()
- **Реквизиты** справочников → доступные атрибуты для вывода

### Шаг 4. Построение и выполнение запроса

1. Пробный запрос: `ВЫБРАТЬ ПЕРВЫЕ 10 ...` — проверить, что имена верны
2. Добавить фильтры и группировки
3. Добавить соединения и вычисления
4. Выполнить через `query-1c.sh` (см. раздел «Выполнение запросов» ниже)

### Шаг 5. Интерпретация и инсайты

- Резюмируй ключевые находки на языке бизнеса
- Выдели аномалии: отрицательные остатки, нулевые продажи, выбросы (значения > 3× среднего)
- Укажи единицы измерения (штуки, рубли, кг)
- Предложи следующие шаги анализа

## Выполнение запросов

Скрипты читают `.env` автоматически и отправляют запрос в 1С.

**macOS/Linux:**
```bash
./.claude/skills/1c-analyst/scripts/query-1c.sh 'ВЫБРАТЬ ПЕРВЫЕ 10 ...'
```

**Windows (PowerShell):**
```powershell
.\.claude\skills\1c-analyst\scripts\query-1c.ps1 -Query 'ВЫБРАТЬ ПЕРВЫЕ 10 ...'
```

Ответ в TSV: первая строка — заголовки, далее данные. При ошибке: HTTP 400 с описанием.

## КРИТИЧЕСКИЕ ОГРАНИЧЕНИЯ ЗАПРОСОВ

### 1. НЕЛЬЗЯ использовать параметры через "&"

```
// НЕПРАВИЛЬНО: ГДЕ Товар = &Товар
// ПРАВИЛЬНО:   ГДЕ Товар.Наименование = "iPhone 17 Pro Max"
```

### 2. ОБЯЗАТЕЛЬНО использовать ПЕРВЫЕ N

Всегда ограничивай количество записей — без лимита запрос может вернуть миллионы строк и вызвать таймаут.

### 3. Сравнение ссылочных полей — только через примитивные реквизиты

```
// НЕПРАВИЛЬНО: ГДЕ Документ.Контрагент = Справочник.Контрагенты.Ссылка
// ПРАВИЛЬНО:   ГДЕ Документ.Контрагент.Наименование = "ООО Ромашка"
// ПРАВИЛЬНО:   ГДЕ Документ.Контрагент.Код = "000001"
```

### 4. Даты — функция ДАТАВРЕМЯ

```sql
ГДЕ Документы.Дата >= ДАТАВРЕМЯ(2026, 1, 1)
    И Документы.Дата < ДАТАВРЕМЯ(2026, 2, 1)
```
Формат: `ДАТАВРЕМЯ(Год, Месяц, День[, Час, Минута, Секунда])`

### 5. Строковые значения — двойные кавычки

```sql
ГДЕ Номенклатура.Наименование = "iPhone 17 Pro Max, 512 Гб"
```

## Синтаксис запросов 1С

### Базовая структура

```sql
ВЫБРАТЬ [РАЗЛИЧНЫЕ] [ПЕРВЫЕ N]
    <Поля выборки>
ИЗ
    <Источники данных>
[ГДЕ
    <Условия отбора>]
[СГРУППИРОВАТЬ ПО
    <Поля группировки>]
[УПОРЯДОЧИТЬ ПО
    <Поля сортировки>]
```

### Источники данных

| Тип объекта | Синтаксис |
|-------------|-----------|
| Справочник | `Справочник.ИмяСправочника` |
| Документ | `Документ.ИмяДокумента` |
| Табличная часть | `Документ.ИмяДокумента.ИмяТабличнойЧасти` |
| Регистр накопления (остатки) | `РегистрНакопления.ИмяРегистра.Остатки()` |
| Регистр накопления (обороты) | `РегистрНакопления.ИмяРегистра.Обороты()` |
| Регистр накопления (остатки и обороты) | `РегистрНакопления.ИмяРегистра.ОстаткиИОбороты()` |
| Регистр сведений | `РегистрСведений.ИмяРегистра` |
| Регистр сведений (срез последних) | `РегистрСведений.ИмяРегистра.СрезПоследних()` |

### Суффиксы ресурсов регистров накопления

**Остатки():** `<Ресурс>Остаток` — например `КоличествоОстаток`

**Обороты():**
- `<Ресурс>Оборот` — оборот (например `КоличествоОборот`)
- `<Ресурс>Приход`, `<Ресурс>Расход` — только для регистров остатков

**ОстаткиИОбороты():**
- `<Ресурс>НачальныйОстаток`, `<Ресурс>КонечныйОстаток`
- `<Ресурс>Приход`, `<Ресурс>Расход`, `<Ресурс>Оборот`

### Псевдонимы

```sql
ВЫБРАТЬ ПЕРВЫЕ 10
    Товары.Код КАК КодТовара,
    Товары.Наименование КАК НаименованиеТовара
ИЗ
    Справочник.Номенклатура КАК Товары
```

### Соединения таблиц

```sql
-- ЛЕВОЕ СОЕДИНЕНИЕ (самое частое)
ВЫБРАТЬ ПЕРВЫЕ 50
    Товары.Наименование,
    ЕСТЬNULL(Остатки.КоличествоОстаток, 0) КАК Остаток
ИЗ
    Справочник.Номенклатура КАК Товары
    ЛЕВОЕ СОЕДИНЕНИЕ РегистрНакопления.ТоварыНаСкладах.Остатки() КАК Остатки
```

`СУММА()`, `КОЛИЧЕСТВО()`, `МАКСИМУМ()`, `МИНИМУМ()`, `СРЕДНЕЕ()`


## Аналитические сценарии

### Анализ продаж

**Топ товаров по продажам**
- Ищи: регистр продаж/реализации (`1c_metadata_search("продажа реализация выручка")`)
- Поля: Номенклатура (группировка), Количество/Сумма (агрегация)
- Запрос: `Обороты()` за период → GROUP BY Номенклатура → ORDER BY Сумма УБЫВ
- Инсайт: сколько позиций дают 80% выручки (правило Парето)

**Динамика продаж**
- Группируй по периодам через `НАЧАЛОПЕРИОДА(Период, МЕСЯЦ)`
- Сравнивай текущий период с предыдущим → рост или падение
- Обрати внимание на сезонность

**Плохо продающиеся товары**
- Стратегия: LEFT JOIN остатков с продажами
- Товары с остатком > 0 и продажами = 0 за N месяцев → неликвид
- Или: товары в нижних 10% по оборачиваемости

**По контрагентам**
- Группируй продажи по Контрагенту → выяви ключевых клиентов
- Сравни с задолженностью — крупный клиент с просроченным долгом = риск

### Складские остатки

**Текущие остатки**
- Быстрая проверка: `1c_register_balances` — покажет остатки + предупредит об отрицательных
- Для детализации: запрос к `Остатки()` с группировкой по складу

**Отрицательные остатки**
- `1c_register_balances` с фильтром на отрицательные значения
- Затем `1c_register_movements` для конкретной позиции — чтобы понять, какой документ создал проблему
- Типичные причины: непроведённый приход, ошибка пересортицы, задвоение расхода

**Неликвиды (dead stock)**
- Товары с остатком, но без продаж за 60-90+ дней
- Стратегия: JOIN `Остатки()` с `Обороты()` за период, отобрать позиции с нулевым расходом

**Оборачиваемость запасов**
- Формула: Средний остаток / Дневные продажи = Дни оборачиваемости
- Используй `ОстаткиИОбороты()` — одна виртуальная таблица содержит и остатки, и обороты
- Норма зависит от отрасли: продукты 7-14 дней, электроника 30-60 дней

### Финансовый анализ

**Маржинальность**
- Ищи регистры с себестоимостью и выручкой
- Маржа = (Выручка - Себестоимость) / Выручка × 100%
- Группируй по товару/категории → найди наиболее и наименее маржинальные

**Дебиторская задолженность** (нам должны)
- Ищи: `1c_metadata_search("взаиморасчет задолженность контрагент")`
- Фильтр: положительный остаток → клиент должен нам
- Группируй по контрагенту, сортируй по сумме УБЫВ
- Красный флаг: долг > 30 дней без движений

**Кредиторская задолженность** (мы должны)
- Тот же регистр, но отрицательный остаток (или наоборот — зависит от конвенции регистра)
- Проверь через `1c_metadata_details` — какой знак означает «мы должны»

### Расследование проблем с данными

**Расхождения между регистрами**
- Сравни остатки из двух разных регистров по одной позиции
- Если не совпадают → ищи документы, которые пишут в один регистр, но не в другой

**Непроведённые документы**
- Запрос: `ГДЕ Проведен = ЛОЖЬ` — документы, которые не создали движений
- Частая причина проблем: «товар списали, а документ не провели»

**Поиск конкретных движений**
- `1c_register_movements` по конкретной позиции → вся история операций
- Помогает найти, когда и каким документом создано движение

**Дубли в справочниках**
- `GROUP BY Наименование HAVING COUNT(*) > 1` — элементы с одинаковым именем
- Часто: один и тот же контрагент или товар заведён несколько раз

### Анализ конфигурации

**Цепочки документов**
- `1c_metadata_search` по типу Document → найти все документы бизнес-процесса
- `1c_metadata_details` для каждого → в какие регистры пишет (поле register_records)
- Восстанови цепочку: Заказ → Поступление → Реализация → Оплата

**Структура регистров**
- `1c_metadata_types` → список всех типов с количеством
- Детализируй каждый регистр → пойми модель данных конфигурации

**Бизнес-логика в коде**
- `1c_code_search("ценообразование")` — как формируются цены
- `1c_code_search("скидка")` — логика скидок
- `1c_code_search("проведение")` — какие движения создаёт документ

### Интеграционные запросы для внешних систем

Когда внешняя система (сайт, мобильное приложение, MES) запрашивает данные из 1С.

**Получение техкарт / рецептур / спецификаций**
- `Document_Спецификация` — спецификации (заголовок: Фирма, Контрагент, СтруктурнаяЕдиница, ИтогоВсего)
- `Document_Спецификация_Состав` — табличная часть (строки спецификации: изделие → материал → количество)
- `Document_Смолл_КалькуляционнаяКарта` — калькуляционные карты (техкарты производства)
- `Document_Смолл_КалькуляционнаяКарта_Состав` — состав калькуляционной карты
- `Document_Смолл_КалькуляционнаяКарта_ТехнологическиеКартыПолуфабрикатов` — полуфабрикаты в техкартах
- Для внешней системы: выбери минимум полей (`$select`), фильтруй по конкретному изделию

**Выгрузка номенклатуры (каталог для сайта)**
- `1c_odata_query` с entity_set `Catalog_Номенклатура`
- Полезные поля: Код, Наименование, Артикул, ЕдиницаИзмерения, ЭтоГруппа, Родитель
- Пагинация: `$top=100` + повторные запросы с `$skip`
- Дополнительно: цены через `InformationRegister_ЦеныНоменклатуры_SliceLast`, остатки через `AccumulationRegister_ТоварыНаСкладах_Balance`

**Заказы товаров**
- Чтение заказов: `1c_odata_query` с `Document_ЗаказПокупателя` или аналогичный
- Табличная часть: отдельный entity_set `Document_ЗаказПокупателя_Товары`
- Фильтрация по дате/статусу: `$filter=Date ge datetime'2026-01-01'`

**Паттерны OData для интеграции**
- Фильтры: `$filter=Наименование eq 'iPhone'`, `$filter=Код eq '00001'`
- Выбор полей: `$select=Ref_Key,Description,Code` (уменьшает трафик)
- Пагинация: `$top=100&$skip=200`
- Сортировка: `$orderby=Date desc`
- Для ссылочных полей OData использует суффикс `_Key`: `Контрагент_Key`, `Номенклатура_Key`

**Рекомендации для внешних API**
- Всегда ограничивай выборку (`$top`) — 1С может отдать миллионы записей
- Кэшируй справочные данные (номенклатура, контрагенты) — они меняются редко
- Остатки и цены запрашивай «по требованию» — они меняются часто
- Используй `Ref_Key` как стабильный идентификатор для связи между системами

## Паттерны аналитических запросов

### ABC-анализ

Цель: классифицировать товары на группы A (80% выручки), B (15%), C (5%).

```
Шаг 1: Получи продажи по товарам, отсортированные по сумме УБЫВ
Шаг 2: Рассчитай накопительный итог — это нужно делать после получения данных,
        т.к. в языке запросов 1С нет оконных функций
Шаг 3: Присвой категорию: A (0-80%), B (80-95%), C (95-100%)
```

### Сравнение периодов

```
Два подзапроса с Обороты() за разные периоды
→ ПОЛНОЕ СОЕДИНЕНИЕ по Номенклатуре
→ ЕСТЬNULL() для обработки товаров, которых не было в одном из периодов
→ Вычисли дельту: Период2 - Период1
```

### Оборачиваемость запасов

```
ОстаткиИОбороты(НачалоПериода, КонецПериода) — одна виртуальная таблица
Средний остаток = (НачальныйОстаток + КонечныйОстаток) / 2
Дневной расход = РасходОборот / КоличествоДней
Дни оборачиваемости = Средний остаток / Дневной расход
```

### Воронка документов

```
Шаг 1: Подсчитай документы каждого типа за период
  - Заказы покупателей: COUNT(*)
  - Реализации: COUNT(*)
  - Оплаты: COUNT(*)
Шаг 2: Конверсия = следующий этап / предыдущий × 100%
  - Заказ → Реализация: сколько заказов отгружено?
  - Реализация → Оплата: сколько отгрузок оплачено?
```

## Частые ошибки аналитика

| Ошибка | Почему это проблема | Как избежать |
|--------|-------------------|-------------|
| Анализ непроведённых документов | Непроведённые документы не создают движений → данные неполные | Фильтр `Проведен = ИСТИНА` |
| Путаница Остатки vs Обороты | Остатки — моментальный снимок, Обороты — за период | Для «сколько сейчас» → Остатки, для «сколько за месяц» → Обороты |
| NULL в LEFT JOIN | Без ЕСТЬNULL() суммы получатся пустыми | `ЕСТЬNULL(Поле, 0)` для числовых полей |
| Неправильный знак оборота | КоличествоОборот = Приход - Расход, может быть отрицательным | Используй суффиксы Приход/Расход отдельно, если важен знак |
| Анализ по документам вместо регистров | Документы — первичные данные, регистры — агрегированные | Для аналитики всегда начинай с регистров |
| Забыл ПЕРВЫЕ N | Запрос без лимита может вернуть миллионы строк → таймаут | Всегда `ВЫБРАТЬ ПЕРВЫЕ N`, начинай с малых значений |
| Использование параметров `&` | HTTP-сервис не поддерживает параметризованные запросы | Значения указывать явно в запросе |
| Сравнение ссылок напрямую | Ссылочные типы нельзя сравнивать как строки | Сравнивать через `.Код`, `.Наименование`, `.ИНН` |
